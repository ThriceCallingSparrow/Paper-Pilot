# Paper Pilot 自动优化系统

## 项目概述

本项目是我们综合课设的内容，很有意思的一个课设形式：原本是通过田口或者部分因子设计完成实验，但我们通过解包flash小游戏在内部添加自动化模拟模块的方式来完成这一点，算是一个很有意思的想法。

欢迎参考、讨论，或基于本项目继续扩展，做出更有意思的项目。

> **PS：**
> 其实老师不希望我们拆开小游戏这个“盒子”去观察里面的构造，可能前几届有人这么做过。
> 但解包后，我们也只是添加了一个自动化模拟用于传递数据，实际的计算逻辑我们并没有去看，依旧是按照黑盒测试的思想去做的。

也希望后面的学弟学妹们看到这个项目后，可以从中获得一些启发，并在本项目的基础上进行拓展和延伸。

## 核心功能

本项目通过 **Python Flask 服务端**、**Optuna 智能优化算法**与 **Flash 游戏端自动化脚本**协同工作，实现纸飞机参数的自动遍历、飞行模拟、结果回传和最优参数搜索。

系统采用任务队列与 HTTP 接口完成跨端通信，形成以下自动化闭环：

```text
生成参数任务 → 游戏端领取任务 → 自动投掷模拟 → 回传飞行结果 → Optuna 更新采样策略 → 继续迭代
```

## 文件说明

| 文件 | 作用 |
|---|---|
| `environment.yml` | 项目运行环境配置。 |
| `paperpilot_optimization.py` | 主优化程序，负责 Flask 服务、任务调度、结果接收及 Optuna 参数寻优。 |
| `untils.py` | 工具文件，用于存放辅助计算函数。 |
| `3flashplayer.exe` | 本地 Flash 播放器。 |
| `3paperpilot.swf` | 原版纸飞机小游戏。 |
| `plane_sim_auto_base.swf` | 自动化飞行模拟程序。 |
| `plane.py` | 使用纯 Python 复刻小游戏计算逻辑的优化程序，运行速度更快。其计算出的飞行距离整体可能高于实际距离，但通常不影响最优参数组合的判断。 |

## 使用方法

1. 先运行后端优化程序：

   ```bash
   python paperpilot_optimization.py
   ```

2. 启动 Flash 自动模拟端：

   将 `plane_sim_auto_base.swf` 拖放到 `3flashplayer.exe` 中运行。

3. 程序开始后，Flash 游戏端会自动向 Python 后端请求任务，执行模拟并回传结果。

> PS：理论上前后端启动顺序颠倒也可以，但推荐先启动 `paperpilot_optimization.py`，再运行 `plane_sim_auto_base.swf`，这样可以避免 Flash 端初始轮询时后端尚未启动导致的请求失败。

## 系统架构

系统由三个主要部分组成：

1. **Python Flask 服务端**
   - 负责维护任务队列。
   - 提供任务获取与结果提交接口。
   - 管理线程锁、超时机制和试验记录。

2. **Optuna 优化模块**
   - 定义参数空间。
   - 使用 `TPESampler` 生成参数组合。
   - 以飞行距离最大化为优化目标。

3. **Flash 游戏自动化端**
   - 自动轮询并领取任务。
   - 修改游戏中的飞机配置。
   - 模拟鼠标拖拽和纸飞机投掷。
   - 监听落地事件并回传结果。

## 通信接口

Python 后端与 Flash 游戏端通过以下接口交互：

### `GET /get_task`

游戏端轮询该接口领取待执行任务。后端返回的信息包括：

- 任务 ID
- 飞机机型与配置
- 鼠标拖拽起点
- 鼠标拖拽终点

### `POST /submit_result`

游戏端完成模拟后，通过该接口提交：

- 任务 ID
- 飞行距离
- 模拟是否成功
- 出手力度
- 飞行角度

项目同时配置跨域支持，以保证 Flash 游戏端能够正常访问 Flask 服务。

## 参数定义与约束

程序预设固定的鼠标拖拽起点，并结合三角函数计算拖拽终点的合法 `X`、`Y` 坐标范围，从而限制鼠标释放位置。

主要优化参数包括：

- 飞机机型：3 种可选
- 机身配重
- 升降舵数值
- 尾翼小翼开关
- 鼠标拖拽终点坐标

后端根据 Optuna 参数空间生成不同的参数组合，并将任务写入全局任务队列，等待游戏端领取。

## 自动模拟流程

1. Flask 后端启动。
2. 后端在独立子线程中运行优化逻辑，主线程持续监听 HTTP 请求。
3. Flash 游戏场景加载完成后，自动化脚本创建日志窗口。
4. 游戏端持续请求 `/get_task`。
5. 获取任务后，游戏端解析任务 ID、飞机配置及拖拽坐标。
6. 调用游戏原生 `resetThrow` 函数重置飞行场景。
7. 自动修改机型、配重、升降舵及小翼配置。
8. 依次调用：
   - `grabPlane`：按下鼠标并抓取飞机
   - `movePlane`：拖拽飞机至指定坐标
   - `releasePlane`：松开鼠标并投掷飞机
9. 游戏原生逻辑根据拖拽距离和角度计算出手力度与飞行角度。
10. 自动化脚本监听飞机落地事件。
11. 飞机落地后，通过 `CalcLength` 计算实际飞行距离。
12. 游戏端调用 `/submit_result` 回传模拟结果。
13. 后端将飞行距离返回给 Optuna，作为本轮目标函数值。
14. Optuna 根据已有试验结果生成下一组参数。
15. 重复以上流程，直至完成全部试验轮次。

## 异常处理与超时机制

系统采用双重超时和异常容错机制，避免单次任务阻塞整个优化流程：

- 后端任务超时时间为 **35 秒**。
- 超时未收到结果时，该任务会被判定为失败，飞行距离记为 `0`。
- 游戏端同样设置 **35 秒**上传超时。
- 接口请求异常或游戏执行报错时，任务会被标记为失败。
- 失败后系统会继续处理下一个任务。
- 后端使用线程锁保护结果字典，避免并发任务造成数据错乱。

## Optuna 寻优逻辑

每轮模拟结束后，后端会记录参数组合及对应的飞行数据，并将飞行距离作为目标函数返回值。

优化设置如下：

- 优化方向：最大化飞行距离
- 采样器：`TPESampler`
- 参数来源：预设参数空间
- 迭代方式：任务生成、模拟、回传与再次采样

全部试验完成后，程序会：

1. 筛选目标值最高的参数组合。
2. 查找该组合对应的完整运行数据。
3. 输出最优机型及配置。
4. 输出最优拖拽坐标。
5. 输出对应的出手力度与飞行角度。
6. 将所有试验明细保存为 CSV 文件。

## 关于 `plane.py` 的说明

`plane.py` 使用纯 Python 复刻小游戏中的飞行计算逻辑，相比 Flash 自动模拟具有更快的计算速度。

目前该程序计算出的飞行距离整体可能高于游戏中的实际距离。该偏差的具体原因尚未确定，但由于参数组合之间的相对优劣关系基本保持一致，因此通常不会影响最优参数值的判断。

## 常见问题说明

### Python 后端收发正常，但 Flash 前端收不到信息

通常情况下，如果 Python 后端日志显示接口收发正常，但 Flash 前端无法收到信息，原因可能是本地沙箱安全策略拦截。

Flash 播放器可能会禁止本地 SWF 发起 HTTP 或 Socket 请求，导致前端无法正常访问 Python Flask 服务端。

解决方法是在 Flash Player 的本地信任目录中添加本地沙箱信任白名单相关文件。Windows 下路径通常为：

```text
C:\Users\用户名\AppData\Roaming\Macromedia\Flash Player\#Security\FlashPlayerTrust\
```

请在该目录下添加对应的信任白名单文件，并将本项目所在目录或相关 SWF 文件路径加入白名单。

> 注意：路径中的 `用户名` 需要替换为当前 Windows 系统用户名。
